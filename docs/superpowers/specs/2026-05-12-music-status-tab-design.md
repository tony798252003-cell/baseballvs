# 音樂狀態 Tab 設計規格

**日期：** 2026-05-12  
**功能：** AdminScreen 新增「音樂狀態」管理 Tab

---

## 背景與目標

目前許多球員的加油歌 URL 未填寫，或雖有填寫但連結已失效，導致播報時無法正常播放。需要一個集中的頁面讓管理者能快速掌握全體球員的音樂狀態，並進行驗證與修正。

---

## 功能範圍

### 1. 新增 Tab

在 AdminScreen 現有的 tab 列（未設定音樂 / 全部球員 / 嗆斯曲）後方新增第四個 tab：**音樂狀態**。

### 2. 球員狀態分組顯示

進入 tab 後，球員依以下三種狀態分組顯示（未驗證前，有 URL 的預設為「待驗證」）：

| 狀態 | 說明 | 顯示樣式 |
|------|------|----------|
| 🔴 連結壞掉 | 有 URL 但驗證失敗 | 紅色標示 |
| ⚫ 沒有歌 | song 欄位空白 | 灰色標示 |
| ✅ 正常 | 驗證通過 | 綠色標示 |

排序優先：連結壞掉 → 沒有歌 → 正常。

### 3. 驗證機制

- 頁面頂部有「全部驗證」按鈕
- 點擊後逐一對每個有 URL 的球員，用靜音 Audio 物件嘗試載入
  - 可播放（`canplay` 事件觸發）→ 標為「正常」
  - 載入錯誤（`error` 事件觸發）→ 標為「連結壞掉」
- 驗證過程中顯示進度：「驗證中 X / N」
- 沒有 URL 的球員不驗證，直接標為「沒有歌」

### 4. 每列操作

每位球員的列包含：
- 姓名、背號、隊伍
- 音樂狀態 badge
- **試播按鈕**：點擊播放 song URL，再點停止；同時只能播一首
- **編輯按鈕**：開啟現有的編輯 modal，修改 URL 後儲存

### 5. 篩選

右上角提供隊伍下拉選單，可篩選只顯示特定隊伍的球員，方便逐隊處理。

---

## 技術細節

- 所有音樂 URL 皆為 Dropbox 分享連結，CORS 限制下無法用 `fetch HEAD` 驗證，改用 Audio 物件載入驗證
- 驗證狀態僅存在前端 reactive state，不寫回資料庫
- 編輯儲存後，該球員狀態重設為「待驗證」

---

## 不在範圍內

- 批次修改 URL
- 自動尋找歌曲來源
- 驗證結果持久化
